# IAM 管理スコープと運用境界

本書は、本リポジトリ（gcp-foundations）の Terraform が **どこまで IAM を管理し、どこからを管理しないか** を明確にするための設計資料である。IAM を意図的に Terraform 管理外に置く運用は十分にあり得るが、その前提をリポジトリ全体で共有しておかないと、後から矛盾（歪み）や事故が生じる。本書はその境界と「守るべき不変条件」を定義する。

______________________________________________________________________

## 1. 管理スコープの定義

本リポジトリの Terraform が管理する IAM は、原則として次の範囲に限定される。

| 区分 | Terraform 管理 | 補足 |
| --- | :--: | --- |
| 組織レベルの**管理者グループ束**（組織管理 / 請求 等） | ✅（`enable_group_iam=true` のときのみ）| `google_organization_iam_member`。グループは Cloud Identity 側で作成済みが前提 |
| 内部ツール用 SA への付与（tfstate/logsink/monitoring 等） | ✅ | `project-iam` モジュール経由。基盤動作に必要な固定的付与 |
| Shared VPC のサービス SA への `networkUser` | ✅（Shared VPC 有効時）| `google_compute_subnetwork_iam_member` |
| **上記以外**（フォルダ/プロジェクト単位、個人ユーザー、顧客担当者、例外付与） | ❌（設計上スコープ外）| 手運用（コンソール）または別途 IaC で管理 |

> つまり、**人間（個人ユーザー）やプロジェクト単位の任意ロール付与を SSoT/Terraform で表現する仕組みは、意図的に持っていない**。`resources` シートの `owner` 列は IAM 付与ではなく**ラベル**である。

## 2. `enable_group_iam` は「管理境界スイッチ」

`enable_group_iam` は単なる on/off ではなく、**IAM をどこで管理するかの境界スイッチ**として理解する。

- **`true`**: 組織レベルの管理者グループへのロール付与を Terraform が管理する。Cloud Identity でグループ（`gcp-organization-admins` 等）が作成済みであることが前提。職務分掌を IaC で統制したい組織向け。
- **`false`**: 組織レベルのグループ IAM を Terraform は**一切付与しない**。これは「ブートストラップ中の一時状態」だけでなく、**小規模・グループ非採用の組織における正規の定常モード**でもある（→ 第4節）。この場合、IAM は手運用（コンソール）または別手段で管理される。

どちらのモードでも、本リポジトリは IAM を**非権威的（additive）**にしか触らないため（第3節）、Terraform 管理外の付与と安全に共存できる。

______________________________________________________________________

## 3. 守るべき不変条件：IAM は必ず非権威的に扱う

手運用の IAM と Terraform を共存させ、歪み・事故を防ぐための**絶対条件**：

> **IAM リソースは必ず非権威的（additive）な `*_iam_member` を使う。権威的な `*_iam_binding` / `*_iam_policy` を組織・フォルダ・プロジェクトのいずれのスコープでも使用しない。**

現状のリポジトリはこれを満たしている（すべて `_iam_member`）：

- `google_organization_iam_member`（`terraform/2_organization`）
- `google_project_iam_member`（`terraform/modules/project-iam`）
- `google_compute_subnetwork_iam_member`（`terraform/modules/app-project-baseline`）

この条件が守られている限り、次が保証される。

- `enable_group_iam=false` のとき、`terraform plan` は IAM について「変更なし」。**コンソールで付与した個人ロールが drift として誤検知されない。**
- `enable_group_iam` を後から `true` にしても、グループ付与は**追加**であり、手で付けた個人ロールを削除しない。

⚠️ **逆に、権威的リソース（`*_iam_binding` / `*_iam_policy`）を導入した瞬間、Terraform が管理対象スコープの IAM を“宣言された状態に強制”し、手運用で付与した権限を消し去る。** これは本設計における最大の事故要因であり、将来にわたって禁止する。

## 4. 小規模・グループ非採用パターン（正規の運用形態）

次のような状況では、Google グループを使わず**個人 ID に直接ロールを付与する**運用が合理的であり、設計上も正規の形態として認める。

- 利用者が少人数（例: 数名規模）で、入退社・異動の回転が小さい。
- 組織の作成ドメインと利用者の実ドメインが異なり、グループ運用がかえって複雑・不透明になる。
- 顧客の意向でグループ運用を採用しない。

このパターンでも、**規模に関係なく守るべき原則**は次のとおり（主体がグループか個人かは本質ではない）。

1. **最小権限**: 個人に `roles/owner` や組織管理者ロールを付けない。対象に必要な範囲ロールのみ付与する。
2. **記録（台帳）を残す**: 付与の事実が後から追える状態にする（第5節の補償統制）。
3. **break-glass を分離**: 管理者級の権限は普段使わない緊急用アカウント1つに限定し、厳重保管する。
4. **定期棚卸**: 半期等で付与状況を見直し、不要権限を剥がす。

> 付与先（グループ/個人）と、付与経路・粒度（記録の有無・最小権限）は**別の軸**である。前者は規模次第で妥協してよいが、後者（記録・最小権限）は規模に関わらず守る。

## 5. 手運用でも監査性を保つ補償統制

IAM を Terraform で管理しない場合でも、監査・説明責任は機能追加なしの運用で担保できる。

- **Cloud Asset Inventory の IAM ポリシー・エクスポート**を集約ログ（logsink）へ定期出力する。「いつ・誰が・何の権限を持っていたか」の証跡が残り、Terraform で“管理”せずとも“記録”は残せる。
- **break-glass アカウント**＋**定期棚卸**で、管理者権限の集中と滞留を抑える。
- 監査要件がある組織では、これらにより「Terraform 管理外だが、台帳・最小権限・監査証跡はある」状態を、小規模に見合った軽さで実現できる。

## 6. グループ採用の判断目安

Google グループ（`enable_group_iam=true`）の価値は、**人の回転・スケール・職務分掌の複雑さ**があって初めて顕在化する。

| 観点 | グループ採用が有利 | 個人付与で十分 |
| --- | --- | --- |
| 利用者数 | 多い／増加見込み | 少数で静的 |
| 入退社・異動 | 頻繁 | ほぼ無い |
| 職務分掌 | 明確に分けたい | 単純 |
| ID ドメイン | 組織ドメインと一致 | 別ドメインで複雑 |

小規模で個人付与を選んだ場合でも、将来スケール・回転が出た時点でグループへ移行できる（個人→グループ付与は追加的に置換可能で、第3節の不変条件により安全に移行できる）。

______________________________________________________________________

## 7. やってはいけないこと（禁止事項）

- 組織・フォルダ・プロジェクトいずれのスコープでも **権威的 IAM リソース（`*_iam_binding` / `*_iam_policy`）を使う**こと（手運用の権限を破壊する）。
- 個人ユーザーに **`roles/owner` / 組織管理者ロールを恒常付与**すること（break-glass を除く）。
- 付与の**記録を全く残さない**コンソール直付与（最低限、第5節の補償統制で記録を残す）。

## 付録: SSoT 駆動 IAM 管理機能について（意図的な未実装）

任意主体（user/group/SA）へ org/folder/project スコープでロールを付与する **SSoT 駆動の IAM 管理機能（例: `iam_bindings` シート）** は、現時点では**意図的に実装していない**。小規模・グループ非採用の組織では過剰であり、テンプレートとしての複雑性を増すだけのためである。

将来、**利用者規模の拡大・人の回転の増加・複数組織での統制要求**が生じた場合に、第3節の不変条件（`_iam_member` のみ）を保ったまま再検討する。
