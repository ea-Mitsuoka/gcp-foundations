# 自動生成エンジンの設計思想 (Hybrid Automation Philosophy)

本リポジトリでは、`terraform/scripts/generate_resources.py` を通じて Terraform コードを自動生成しています。このエンジンは、単にコードを書き出すだけでなく、**「Terraform の限界を Python で補完し、Terraform の強みを最大限に引き出す」** というハイブリッドな設計思想に基づいています。

## 1. なぜ Python で生成するのか？

Terraform は「宣言的」な言語であり、静的な構成管理には非常に強力ですが、以下のケースではコードが複雑化し、メンテナンスコストが急増します。

1.  **動的な階層構造**: フォルダの親子関係が任意に深くなる場合、Terraform の `for_each` だけで依存関係を解決するのは困難です。
2.  **複雑な文字列操作**: 命名規則のバリデーションやサニタイズは、Python の方が柔軟かつ確実に実行できます。
3.  **SSOT との橋渡し**: Excel (SSOT) のデータを直接 Terraform で読み込むよりも、一度 Python でパースして「正しいデータ型」に変換する方が、Terraform 側のコードがシンプルになります。

## 2. 三つの生成パターンとその使い分け

リソースの特性に応じて、以下の3つの生成パターンを使い分けています。

| パターン | 生成物 | 主な対象リソース | 選択の理由 |
| :--- | :--- | :--- | :--- |
| **A. 直接 TF 生成** | `.tf` ファイル | フォルダ (L3), VPC-SC (L2) | **構造の複雑性への対応**。無限の階層構造や、複雑なリソース間参照を Python 側で確定的なコードとして書き出すため。 |
| **B. tfvars 生成** | `terraform.tfvars` | プロジェクト (L4) | **ロジックとデータの分離**。実績のある共通モジュール（Project Factory）に対し、プロジェクト固有の「設定値」のみを流し込むため。 |
| **C. CSV 生成** | `.csv` ファイル | サブネット (L1), アラート定義 (L1) | **大量同質データの効率化**。Terraform の `csvdecode` を活用し、Terraform 側のコードを極限まで DRY（重複なし）に保つため。 |

## 3. 開発者へのメッセージ

このエンジンをメンテナンス、または他プロジェクトへ移植する際は、以下の原則を守ってください。

### ロジックはモジュールに、データは生成物に
Terraform 側に複雑な `dynamic` ブロックや高度な関数を詰め込みすぎないでください。Terraform コードが読みにくくなったら、それは Python 側でデータを整理すべきサインです。

### バリデーションの徹底
`terraform apply` を実行する前にエラーを検知できるよう、Python 側で徹底的にバリデーション（重複チェック、型チェック、親子関係チェック）を行ってください。

### 冪等性の担保
`random_id` などのランダム要素を避け、SSOT（Excel）から常に同じ Terraform コードが生成されるように設計してください。これにより、IaC の最大のメリットである「再現性」が担保されます。

---

この設計は、エンジニアの「思考の整理」を助け、大規模なクラウドインフラを少人数で安全に管理するための武器となります。
