# 環境の一括解体・クリーンアップガイド (Environment Destruction)

本ドキュメントでは、GCP Foundations 基盤で展開されたリソース群を、依存関係を壊すことなく安全かつ一括で削除するための `make destroy` コマンドの仕様と運用方針について解説します。

______________________________________________________________________

## 1. コマンドの基本仕様

本基盤は複数の Terraform レイヤー（L0〜L4）で構成されています。個別のディレクトリで `terraform destroy` を実行すると依存関係エラー（デッドロック）が発生する可能性があるため、必ずルートディレクトリで `make destroy` を使用してください。

### 🚨 最重要ルール：設計図（Excel）は解体するまで消さない！

Terraformがリソースを正しく解体するには、生成された構成ファイル（`.tf`）が必要です。
**環境を解体する前に、`gcp-foundations.xlsx` からプロジェクトの行を削除しないでください。** 行を削除して `make generate` を実行してしまうと、設計図が消滅して解体が実行できなくなります。
**「① 環境を解体する ➔ ② Excelから消してコードを再生成する」** という順番を必ず守ってください。

### 環境の解体手順

**ステップ1: 破壊的変更の許可とロック解除の適用**
`terraform/common.tfvars` を編集し、全リソースのロックを解除するフラグを立てます。

```hcl
allow_resource_destruction = true
```

次に、実行モード（テストモードのON/OFF）に応じて以下の対応を行います。

- **本番モード (テストモードOFF):** **必ず `make deploy` を実行してください。** GCP上のリソースおよびStateに対してロック解除設定（`deletion_protection = false`）を反映させる必要があります。これを怠ると、`make destroy` は保護エラーで安全に停止します。
- **テストモード (テストモードON):** **個別の `make deploy` は不要です。** `make destroy` スクリプト内部で、破壊の直前に自動的にロック解除の Apply が実行されます。

**ステップ2: 解体の実行**
ロック解除の準備が整ったら、以下の解体コマンドを実行します。

```bash
# L2〜L4のみを解体する場合（日常的なリセット）
make destroy

# L1管理プロジェクトも含めて完全に更地にする場合
make destroy ALL
```

## 2. 削除範囲の比較まとめ (オプションの違い)

`make destroy` は、付与するオプションによって削除対象となるレイヤーやリソースの範囲（深さ）を柔軟に制御できます。

| 対象ディレクトリ (レイヤー) | 役割 | `make destroy`<br>(引数なし) | `make destroy LAYER=n`<br>(例: LAYER=3) | `make destroy ALL`<br>(ALLオプション) |
| :--- | :--- | :---: | :---: | :---: |
| **`4_projects/` (L4)** | 各種アプリ・システムのプロジェクト | 🗑️ 削除される | 🗑️ 削除される | 🗑️ 削除される |
| **`3_folders/` (L3)** | フォルダ階層 | 🗑️ 削除される | 🗑️ 削除される | 🗑️ 削除される |
| **`2_organization/` (L2)** | 組織ポリシー・組織タグ | 🗑️ 削除される | 🛡️ 維持される | 🗑️ 削除される |
| **`1_core/services/` (L1)** | 管理プロジェクト内の設定（NW、データセット等） | 🗑️ 削除される | 🛡️ 維持される | 🗑️ 削除される |
| **`1_core/base/` (L1)** | 管理プロジェクトそのもの（器） | 🛡️ **維持される** | 🛡️ 維持される | 🗑️ **削除される** |
| **`0_bootstrap/` (L0)** | tfstate保存バケット等の最下層基盤 | 🛡️ 維持される | 🛡️ 維持される | 🛡️ 維持される |

## 3. 各オプションの運用ユースケース

### ① `make destroy` (引数なし)

**用途: 日常的な環境リセット・再構築**
L1の「管理プロジェクト（器）」を残し、中身のサービス設定や上位のアプリ・フォルダ階層をすべて更地にします。
器を残すことで、再構築時にGCP特有の「課金アカウントのリンク」の手間をスキップし、プロジェクトIDの枯渇を防ぎながら高速にインフラを最新化できます。

### ② `make destroy ALL`

**用途: インフラの完全な作り直し**
管理プロジェクトの器も含めて、インフラ全体を白紙に戻したい場合に使用します（※L0のtfstate基盤は安全のため残ります）。実行後は、次にデプロイする際に再度手動（またはAPI）での課金アカウントリンク作業が発生します。

### ③ `make destroy LAYER=n`

**用途: 特定の階層のみのロールバック**
指定したレイヤー（L4〜Ln）のみを削除します。例として `make destroy LAYER=3` を実行すると、L4(アプリ)とL3(フォルダ)だけが削除され、L2の組織ポリシーやL1のネットワーク基盤はそのまま維持されます。

______________________________________________________________________

## 4. 特殊な運用シナリオ (Advanced Usage)

### 組織ポリシー (L2) を維持したまま、管理プロジェクト (L1 Base) を削除する

**コマンド: `make destroy LAYER=3 ALL`**

「組織全体のガードレール（L2: 組織ポリシー）は一度定義したら動かしたくないが、管理プロジェクト（L1: Base）のプロジェクトIDや請求先との紐付けをやり直したい」という極めて限定的な状況で有効な「裏技」的な使い方です。このコマンドを実行すると、L4（アプリ）とL3（フォルダ）が削除された後、L2（組織ポリシー）とL1-Services（ネットワーク設定等）の削除プロセスをスキップし、最後にL1-Base（管理プロジェクトの器）のみを解体します。

## 5. テストモード (`make test-mode`) 時の挙動

テストモードがON（`.test_mode_env` が存在し、Prefixにサフィックスが付与されている状態）の場合、`make destroy` は検証の効率化のために以下の特殊な挙動を示します。

### ① 自動ロック解除機能

テストモード時は、スクリプトが各レイヤーの解体直前に自動で `terraform apply` を実行し、削除保護を無効化します。運用者は `make deploy` を明示的に実行することなく、一気通貫で環境を解体できます。

### ② 管理プロジェクト (L1) の自動保護

テストモード時は、構築に時間がかかる `logsink` および `monitoring` 関連のプロジェクト・リソースは、たとえ `make destroy ALL` を実行したとしても\*\*自動的にスキップ（維持）\*\*されます。これにより、検証用のアプリプロジェクトやフォルダ階層だけを高速にスクラップ＆ビルドすることが可能です。
