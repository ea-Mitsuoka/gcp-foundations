# スプレッドシート・ワークショップ・ガイド (Live Session Guide)

本ドキュメントは、顧客と画面共有をしながら `gcp-foundations.xlsx` (SSoT) を完成させるためのファシリテーション・ガイドです。
単に Excel を埋める作業ではなく、**「顧客のビジネス要件を GCP の堅牢な構造にどう翻訳するか」** という意思決定をサポートするためのステップを詳述します。

______________________________________________________________________

## 1. セッションの全体像とゴール

- **ゴール**: `gcp-foundations.xlsx` の全シートが埋まり、`make generate` でエラーが出ない状態にすること。
- **マインドセット**: 「今の設計が、将来の拡張性とセキュリティの土台になる」ことを顧客に伝え、納得感を持って進めます。

## 2. ワークショップ・ステップ (詳細ガイド)

### Phase 1: 組織図と統制タグの設計 (`resources` + `tag_definitions` シート)

**目的**: 管理の境界線を引くと同時に、組織ポリシーの条件としても使える強力な「タグ」を定義します。

- **問いかけの例**:
  - 「本番環境と開発環境で、管理する担当者は分かれていますか？」
  - 「コストセンターやシステム分類など、組織を横断する管理軸はありますか？」
- **設計のヒント (1) - `tag_definitions`シート**:
  - まず、組織で利用したい「タグのキー（例: environment, cost_center）」と、その「選択肢（例: production, development）」を定義します。
  - ここで定義したもの以外は使えないため、タイポを防ぎ、ガバナンスを強制できます。
- **設計のヒント (2) - `resources`シート (フォルダ編)**:
  - `tag_definitions`で定義した内容を元に、フォルダ構造を設計します。
  - `resource_type` に `folder` を指定し、`parent_name` で親子関係を定義します。
  - `org_tags` 列に `environment/production` のように、定義済みのタグを割り当てていきます。

### Phase 2: 基盤ネットワークの設計 (`shared_vpc_subnets` シート)

**目的**: ネットワーク管理を中央集約し、安全でクリーンな通信環境を構築します。

- **問いかけの例**:
  - 「IPアドレスの管理を各プロジェクトに任せて、範囲が重複するリスクを許容できますか？」
  - 「オンプレミスとの接続は予定していますか？」
- **設計のヒント**:
  - **Shared VPC (推奨)**: ネットワーク管理を「インフラチーム」に集約。各プロジェクトには「サブネットの利用権」だけを払い出します。
  - **host_project_env**: `prod` (本番) と `dev` (開発) でホストプロジェクトを分けることで、ネットワークレベルでの物理的な分離を担保します。
- **Excel への入力**:
  - `subnet_name`: `prd-subnet-01` のように、用途と環境がわかる名前を定義。
  - `ip_cidr_range`: 重複しないプライベートIP範囲を指定。

### Phase 3: セキュリティ境界の設計 (`vpc_sc_perimeters` / `access_levels` シート)

**目的**: 機密データを扱うプロジェクトを「物理的な壁」で囲み、外部流出を防止します。

- **問いかけの例**:
  - 「絶対に外部へ流出してはいけない機密データ（個人情報、機密ドキュメント等）を扱う予定はありますか？」
- **設計のヒント**:
  - **Perimeter (境界)**: 境界内のプロジェクト間は通信可能ですが、境界外（インターネット等）へのデータ持ち出しを API レベルでブロックします。
  - **Access Level**: 「特定のオフィスIPからのみ境界内へのアクセスを許可する」といった例外ルールを定義します。
- **Excel への入力**:
  - `restricted_services`: 保護したいサービス（`storage.googleapis.com` 等）を選択。

### Phase 4: ガードレールの設計 (`org_policies` シート)

**目的**: 意図しない設定（不注意によるパブリック公開など）をシステム的に防止します。

- **問いかけの例**:
  - 「VMインスタンスにパブリックIPを付与することを原則禁止しますか？」
  - 「リソースの作成場所を日本国内（東京/大阪）に限定しますか？」
- **設計のヒント**:
  - **段階的適用**: 移行プロジェクトの場合は、最初は `enforce = FALSE` で定義し、移行完了後に `TRUE` に切り替える「ソフトランディング」を提案してください。
- **Excel への入力**:
  - `target_name`: 組織全体なら `organization_id`、特定のフォルダのみならそのフォルダ名を指定。

### Phase 5: ログ集約戦略の設計 (`log_sinks` シート)

**目的**: 組織の全ログを一元的に集約し、監査証跡の確保とセキュリティ分析の基盤を構築します。

- **問いかけの例**:
  - 「監査ログは最低何年間、保管が必要ですか？」
  - 「長期間保管するログは、コストの安いストレージに自動で移動させたいですか？」
- **設計のヒント**:
  - ログの種類（`log_type`）ごとに、宛先（`destination_type`: BigQuery or Cloud Storage）と保持期間（`retention_days`）を定義します。
  - 例えば、「監査ログはBigQueryに365日保管し、それ以降はアーカイブ」といった要件を実装できます。
- **Excel への入力**:
  - `filter` に `protoPayload.methodName:*` などの抽出条件を定義します。

### Phase 6: プロジェクトの展開 (`resources` シート: プロジェクト編)

**目的**: これまでの設計を各アプリケーションに適用し、リソースを払い出します。

- **問いかけの例**:
  - 「このプロジェクトは、先ほど設計したどのネットワーク、セキュリティ境界、統制タグを使いますか？」
- **設計のヒント**:
  - `resource_name`: `prd-app-01` のように、`環境-用途-連番` の命名規則を推奨。
  - `owner` 列に担当者メールアドレスを記入し、責任の所在を明確にします。
- **Excel への入力**:
  - `resource_type` を `project` に設定します。
  - `shared_vpc`, `vpc_sc`, `org_tags` の各列に、前フェーズで定義した名前を入力して、プロジェクトを中央の仕組みに紐付けます。

### Phase 7: 監視と通知の設計 (`notifications` / `alert_definitions` シート)

**目的**: 異常事態を検知した際、誰にどう伝えるかを自動化します。

- **問いかけの例**:
  - 「Phase 5で集約したログの中から、『エラー』という単語が記録されたら、どのメールアドレスに通知を飛ばすべきですか？」
- **設計のヒント**:
  - `alert_name` (`error_log_alert`など) を共通キーとして、通知先（`notifications`シート）とアラート条件（`alert_definitions`シート）を紐付けます。
- **Excel への入力**:
  - `metric_filter`: `severity="ERROR"` などのログフィルタ条件を定義します。

## 3. ライブ入力中のファシリテーション・テクニック

1. **「なぜ？」をセットで語る**:
   - 「ここを `TRUE` にしますね」ではなく、「セキュリティリスクを排除するために、ここでは外部IPを禁止する設定（TRUE）にしておきますね」と、意図を添えます。
1. **プルダウンを活用する**:
   - 入力ミスは後のデプロイ失敗に直結します。「Excelのプルダウンから選んでください」と徹底させることで、手戻りを防ぎます。
1. **その場でコードを生成してみせる**:
   - ある程度入力が終わったら、その場で `make generate` を実行します。
   - 「今決めた設定が、これだけのTerraformコード（数千行になることもあります）に一瞬で変換されました」と見せることで、自動化の価値を体感させます。

## 4. セッションの締めくくり

1. **整合性チェックの実行**:
   - `make generate` が成功することを確認し、赤いエラーが出ていないことを顧客と一緒に確認します。
1. **プランの提示**:
   - `terraform plan` の結果を見せ、何が作られようとしているかを最終確認します。
1. **次ステップへの合意**:
   - 構成に間違いがないことを確認し、実際のデプロイ（`make deploy`）への承認を得ます。
