# GCP Foundations テスト計画書

## 1. はじめに

### 1.1. 目的

本ドキュメントは、`gcp-foundations`リポジトリにおけるIaC基盤の品質、セキュリティ、および運用の信頼性を多角的に保証するためのテスト戦略を定義する。
従来の単体テストに加え、生成されたコードの妥当性、GCP組織全体へのガバナンス適用、およびレイヤー間の依存関係を網羅的に検証することを目的とする。

### 1.2. 範囲

- **対象**:
  - `gcp-foundations.xlsx` (SSoT) のバリデーションロジック。
  - 自動生成されたTerraform構成ファイル (`auto_*.tf`, `tfvars`)。
  - Terraformモジュール (`terraform/modules/`)。
  - 各デプロイレイヤー (L0〜L4)。
  - OPA (Rego) によるガバナンスポリシー。
  - GitHub Actionsによる自動化ワークフロー。

## 2. テスト戦略とテストレベル

本プロジェクトでは、ピラミッド型のテスト構造を採用し、早い段階（左側）でバグを検知する「シフトレフト」を実践する。

### 2.1. 静的解析 (Static Analysis) - 【CIで毎実行】

コードを実際にデプロイすることなく、構文エラーやベストプラクティス違反を検知する。

- **Terraform**: `fmt`, `validate`, `tflint` による構文とスタイルのチェック。
- **Security**: `checkov` による不当な公開設定や脆弱な構成の検知。
- **Policy (OPA)**: `make opa` による、独自定義されたガードレール（ラベル必須化等）の検証。
- **Python**: `ruff` によるコード品質と型のチェック。

### 2.2. 単体テスト (Unit Testing) - 【CIで毎実行】

個々のコンポーネントが単体で正しく動作することを検証する。

- **Logic**: `pytest` を用いた `ResourceValidator` の全パス検証（正常・異常）。
- **Modules**: Terraform Test Framework (`.tftest.hcl`) を用い、各共通モジュールが意図した属性のリソースを生成するかを検証。

### 2.3. 結合テスト (Integration Testing) - 【ステージング環境で実行】

コンポーネント間、およびTerraformレイヤー間の相互作用を検証する。

- **SSoT-to-Code**: 仮想的なExcel定義から `make generate` を実行し、生成された `.tf` ファイルが期待される構造（ファイルパス、リソース名）を持っているかを検証。
- **Cross-Layer Dependency**: 上位レイヤー（例: L3 Folder）の出力（フォルダID）が、下位レイヤー（例: L4 Project）の入力として正しく渡され、リソースが紐付くかを確認。

### 2.4. システム / E2Eテスト (System/E2E Testing) - 【テスト用組織で実行】

実際のGCP環境にデプロイし、機能横断的なユースケースを検証する。

- **Infrastructure Deployment**: `make deploy` による一括構築。
- **Governance Verification**: 組織ポリシーやVPC-SCが、実際にAPIリクエストをブロックするかを確認。
- **Operational Scenarios**: 「ログベースアラートの発報」など、運用フローをエンドツーエンドで再現。

## 3. 網羅的な検証項目

### 3.1. SSoT・自動生成の網羅性

- [ ] **バリデーション全件**: CIDR重複、階層の循環、命名規則違反がすべて検知されるか。
- [ ] **全リソースタイプ**: フォルダ、プロジェクトに加え、タグ、VPC-SC、組織ポリシーがすべて正しく出力されるか。
- [ ] **冪等性**: Excelに変更がない場合、`make generate` および `terraform plan` で差分が発生しないか。

### 3.2. ガバナンス・セキュリティの網羅性

- [ ] **組織ポリシー**: `enable_org_policies` の ON/OFF が各レイヤーに正しく反映されるか。
- [ ] **VPC-SC**: 境界（Perimeter）へのプロジェクト追加・削除が、期待通りにAPIを保護/解放するか。
- [ ] **IAM**: 簡略構成（2グループ）と標準構成（9グループ）が、意図したロールセットを付与するか。

### 3.3. 異常系・エッジケース

- [ ] **Quotaエラー**: プロジェクト作成上限に達した場合のスクリプトの挙動とエラーハンドリング。
- [ ] **API依存関係**: 前提となるAPIが未有効化の状態でデプロイした際、適切なエラーが表示されるか。
- [ ] **課金未リンク**: 課金アカウントが紐付いていない場合に、API有効化レイヤーが安全にスキップされるか。

## 4. 品質指標 (Quality Metrics)

以下の指標を目標値として設定し、品質を可視化する。

| 指標 | 目標値 | 確認タイミング |
| :--- | :--- | :--- |
| **バリデーション・カバレッジ** | 100% (ResourceValidatorの全ロジック) | 月次レビュー |
| **OPA ポリシー合格率** | 100% (PRマージの必須条件) | CI実行毎 |
| **デプロイ成功率** | 95%以上 (手動変更によるドリフトがない状態) | 週次 (Drift Check) |
| **テスト自動化率** | 計画項目の 70% 以上 | 四半期毎 |

## 5. テスト環境とデータ

- **組織**: `gcp-foundations-test-org` (検証用)
- **データ**: `tests/fixtures/` 配下にシナリオ別のExcelファイルを整備し、データ駆動型でテストを実施する。
- **クリーンアップ**: テスト終了後は、`make destroy` を用いて全レイヤーのリソースを削除し、残存リソースによるコスト発生を防ぐ。

### 5.1. リソースの完全削除 (make destroy)

本基盤は複数のTerraformレイヤーで構成されているため、個別に `terraform destroy` を実行するのではなく、依存関係を考慮して一括削除を行う `make destroy` の使用を推奨する。

**注意事項:**

- **破壊許可フラグ**: 誤操作防止のため、`terraform/common.tfvars` 内の `allow_resource_destruction` が `true` に設定されていない場合、コマンドは失敗する。
- **実行モード**:
  - `make destroy`: アプリプロジェクト、フォルダ、組織ポリシー、共有VPC等の「中身」を削除する。
  - `make destroy --all`: 上記に加え、管理用プロジェクト（Seedプロジェクト等）自体も削除対象に含める。
- **手動確認**: 実行時に `DESTROY` という文字列の入力を求められる。

### 5.2. ローカル環境の初期化 (make clean)

環境を完全にリセットしたい場合や、別の環境（プロジェクト）へのデプロイに切り替える際は、`make clean` を実行する。
このコマンドは以下のローカルファイルを削除する。

- `.terraform/` ディレクトリ（バイナリおよびローカルキャッシュ）
- `.terraform.lock.hcl`（プロバイダのロックファイル）
- `__pycache__`（Pythonのキャッシュ）

※ `make clean` はGCP上の実リソースには影響を与えず、ローカルの作業状態のみをクリアする。
