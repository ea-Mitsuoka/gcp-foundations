# GCP Foundations テスト設計書

## 1. はじめに

### 1.1. 目的

本ドキュメントは、[テスト計画書](./テスト計画書.md)で定義されたテスト項目に基づき、`gcp-foundations`リポジトリのテストを「どのように」実施するかの具体的なアプローチと設計を定義する。

### 1.2. 参照ドキュメント

- [GCP Foundations テスト計画書](./テスト計画書.md)
- [アーキテクチャ設計書](./docs/design/architecture.md)
- [スプレッドシートの仕様書](./docs/setup/spreadsheet_format.md)

## 2. テスト設計方針

### 2.1. テスト種別

本リポジトリのテストは、コードやインフラ構成といった内部構造を理解した上で行う**ホワイトボックステスト**を主軸とする。これにより、特定のモジュールやスクリプトの動作をピンポイントで検証する。
`make deploy`の実行やエンドツーエンドの機能検証など、システム全体を外部から操作するテストにおいては、**ブラックボックステスト**のアプローチも併用する。

### 2.2. テスト自動化方針

- **自動化済み**:
  - **単体テスト**: Pythonのバリデーションロジック (`pytest`)。
  - **静的解析**: Terraform, Python, Shell, RegoのLint/フォーマットチェック (`make lint`, CI)。
- **手動テスト**:
  - 本書で設計する結合テスト、システムテスト、E2Eテストは、初回は手動での実行を基本とする。
  - **将来的な展望**: [テスト計画書](./テスト計画書.md)の「4. 今後の課題」に記載の通り、手動テストで有効性が確認された項目から、TerratestやPlaywright（コンソール操作）等を活用した自動化を推進する。

### 2.3. テストデータ管理

テストシナリオの再現性と効率性を確保するため、テストデータを体系的に管理する。

- **SSoT (Excel/CSV)**:
  - `tests/fixtures/`ディレクトリ（新規作成を想定）配下に、各テストシナリオ用の`gcp-foundations.xlsx`およびCSVファイルを配置する。
    - `happy-path.xlsx`: 全ての機能が正常に動作する標準的な設定ファイル。
    - `validation-error-duplicate-project.xlsx`: プロジェクト名が重複する異常系データ。
    - `validation-error-cidr-overlap.xlsx`: サブネットのCIDRが重複する異常系データ。
  - テスト実行前に、対象のフィクスチャファイル（テストデータ）をリポジトリのルートにコピーして使用する。

## 3. テスト環境設計

- **GCP環境**:
  - テスト専用に払い出された、クリーンなGCP Organizationを使用する。これにより、既存リソースとの予期せぬ競合や権限問題を完全に排除する。
- **実行環境**:
  - [ローカル開発環境セットアップガイド](./docs/development/local_development.md)に記載のツール群がすべてインストールされたクリーンな環境（DockerコンテナやクリーンなVMを推奨）。
- **テスト用ID**:
  - IAMやVPC-SCのテストを正確に行うため、以下のGoogleアカウントおよびグループを事前にテスト用Workspace/Cloud Identityで作成しておく。
    - `test-org-admin@<test-domain>`
    - `test-developer@<test-domain>`
    - `gcp-developers@<test-domain>` (グループ)
  - テスト実行者は、これらのテスト用ユーザーに切り替わりながら操作を行う。

## 4. 機能エリア別テスト設計

[テスト計画書](./テスト計画書.md)のテスト項目に対する、具体的なテストアプローチを定義する。

### 4.1. SSoT とコード生成 (TC-GEN)

- **アプローチ**: `make generate`スクリプトの入力（Excel/CSV）と出力（標準出力/終了コード）に着目した、データ駆動テスト。
- **設計**:
  - **異常系 (TC-GEN-01)**: 上記「2.3. テストデータ管理」で準備した異常系データ（例: `validation-error-duplicate-project.xlsx`）をリポジトリルートにコピーし、`make generate`を実行。スクリプトが`exit 1`で終了し、期待されるエラーメッセージが出力されることを確認する。
  - **正常系 (TC-GEN-02, 03)**: `happy-path.xlsx`を用いて`make generate`を実行。成功することを確認後、`git diff`を用いて生成された`auto_*.tf`ファイルの内容が、Excelの定義と一致しているかを目視でレビューする。

### 4.2. 初期構築とライフサイクル (TC-LC)

- **アプローチ**: 実際の運用担当者の視点に立った、シナリオベースのシステムテスト。
- **設計**:
  - `initial_setup.md`および`project_lifecycle.md`の各手順を、テスト担当者がドキュメントのみを頼りに実行する。
  - **課金未設定 (TC-LC-03)**: `deploy_all.sh`のログを精査し、`Skipping ... (core_billing_linked is false)`というメッセージが出力され、`1_core/services`配下のディレクトリが意図通りにスキップされることを確認する。
  - **リソース削除 (TC-LC-05)**: `terraform apply -destroy -target=...`実行後に`terraform plan`を実行し、対象プロジェクト以外のリソースに意図しない差分（`destroy`など）が発生していないことを確認する。

### 4.3. セキュリティ & ガバナンス (TC-SEC)

- **アプローチ**: 「侵入者」「設定ミスをする運用者」の視点に立った、攻撃ベース（アタックシナリオ）のテスト。
- **設計**:
  - **組織ポリシー/VPC-SC (TC-SEC-01, 02, 03)**:
    1. `make deploy`でセキュアなベースラインを構築する。
    2. テスト用ユーザー (`test-developer`) として`gcloud`に再ログインする。
    3. ポリシーに違反する操作（例: SAキー作成、許可外リージョンへのリソース作成、境界外へのデータ持ち出し）を`gcloud`コマンドまたはGCPコンソールから試行する。
    4. GCPから返されるAPIエラーメッセージが、意図したポリシー違反によるものであることを確認する。
  - **IAMモデル (TC-SEC-04)**: `test-developer@<test-domain>`を`gcp-developers`グループに追加後、同ユーザーでログインし、プロジェクトの閲覧（許可）と編集（拒否）の両方を試みる。

### 4.4. E2E機能テスト (TC-E2E)

- **アプローチ**: 複数のコンポーネント（GCPサービス、Terraformリソース）が連携して動作する機能を、エンドユーザーの視点で検証する。
- **設計**:
  - **非アクティブアカウント監視 (TC-E2E-01)**:
    1. テストの実行時間を短縮するため、`inactive_users_view.sql`内の日付比較 (`INTERVAL 90 DAY`) を `INTERVAL 1 MINUTE` に変更し、`make deploy`で適用する。
    2. テストユーザー (`test-e2e-user`) を作成し、何らかの閲覧権限を付与する。
    3. 数分間待機する。
    4. Cloud Schedulerコンソールからジョブを手動で実行する。
    5. Cloud Monitoringでカスタム指標の値が`1`になること、および最終的にメール通知が届くことを確認する。
  - **ログベースアラート (TC-E2E-02)**: `gcloud logging write`コマンドを用いて、テスト対象のアラート定義（`alert_definitions.csv`）が検知すべきログメッセージを意図的に生成し、通知の受信までを確認する。
