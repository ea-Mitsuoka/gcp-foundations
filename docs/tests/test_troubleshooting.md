# テスト実行時トラブルシューティング・ガイド (Test Troubleshooting)

本ドキュメントは、GCP Foundations 基盤のテスト実行中（および今後の実運用中）に発生したエラーと、その具体的な解決手順を記録したナレッジベースです。
エラーに遭遇した場合は、まず本ドキュメントの表から類似の症状を探し、記載されている解決策を試してください。

______________________________________________________________________

## エラー解決ログ

| 発生フェーズ | エラーメッセージ / 症状 | 原因 | 解決策 (実行コマンド) | 備考 |
| :--- | :--- | :--- | :--- | :--- |
| `make deploy`<br>`make destroy` | `oauth2: "invalid_grant" "reauth related error (invalid_rapt)"` | Terraformが裏側で使用している「Application Default Credentials (ADC)」の認証トークンの有効期限が切れた（またはGoogle Workspaceのセッション再認証要求に引っかかった）ため。 | ターミナルで以下の2つのコマンドを実行し、ブラウザで再認証を行う。<br><br>`gcloud auth login`<br>`gcloud auth application-default login` | Pre-flight checkが通過（表面的なgcloud認証は有効）していても、Terraform用のAPIトークン(ADC)が切れていると発生する。長時間の作業時に遭遇しやすい。 |
| `make deploy` | `Error creating MonitoredProject: googleapi: Error 403: The caller does not have permission` <br>*(※20分近く `Still creating...` で固まった後に発生)* | 【解決済】旧アーキテクチャ（L1一括Scoping）における初期化遅延とポーリングのデッドロック。現在はL4モジュールからの即時登録（Push型）へアーキテクチャを変更したため発生しません。 | L4 `app-project-baseline` 内でのPush型設計に移行したことで根本解決済み。対応不要。 | GCP側のAPI仕様/遅延に起因するため、Terraform側からの根本解決が難しい既知の罠。L1の監視連携部分なので、スキップしてもL3/L4リソースの構築テストには影響しない。(例:`mv terraform/1_core/services/monitoring/scoping/main.tf terraform/1_core/services/monitoring/scoping/main.tf.bak`, `mv terraform/1_core/services/monitoring/scoping/outputs.tf terraform/1_core/services/monitoring/scoping/outputs.tf.bak`) |
| `make deploy` | `Error 400: field [parent] has issue [Parent id must be numeric.]` | プロジェクト作成時、親（フォルダまたは組織）のIDとして数字のみが要求される箇所に対し、`"organizations/123456789"` のような文字列が渡された、もしくは自動生成の残骸（文字列）が残っていたため。 | 1. 過去のテストディレクトリの残骸を手動で削除する（`rm -rf terraform/4_projects/不要なディレクトリ`）。<br>2. Terraformモジュール側 (`project-factory/main.tf`) で `replace(var.organization_id, "organizations/", "")` のように不要な文字列を取り除く処理を追加し、堅牢化する。 | Terraformの `data.google_organization` が返すIDには `"organizations/"` が付与される仕様があり、APIの要求する純粋な数値フォーマットとズレることで発生するGCP Terraformあるあるの罠。 |
| `make deploy` | `googleapi: Error 409: Requested entity already exists` | 1. 手動で「シャットダウン」したプロジェクトIDが、30日間の「削除待ち」状態で予約されている。<br>2. `gcp-foundations.xlsx` から削除したはずのプロジェクト定義が、ファイル内に意図せず残ったまま `make generate` された。<br><br>**【ログからの特定】**<br>`make generate` のログに `⚠️ WARNING: Project ... was removed` という**警告が出ていない**場合、これが原因です。 | 1. `gcp-foundations.xlsx` を開き、対象プロジェクトの行が完全に削除されていることを確認し、上書き保存する。<br>2. `make generate` を実行し、`⚠️ WARNING...` の警告が出ることを確認する。<br>3. `make deploy` を再実行する。 | GCPプロジェクトIDは30日間再利用不可。テスト等で作り直す際は、`proj-v2` のように別名を付けるのが定石。`deploy_all.sh` は `4_projects` 配下のディレクトリを読み込むため、`generate` 時に定義が残っていると必ず再作成を試みます。<br><br>テスト用スクリプト (`python3 terraform/scripts/toggle_test_mode.py`) を実行してテストモードを【ON】にしてください。ランダムなプレフィックスが付与され、不要な管理プロジェクト（logsink等）の構築も自動でスキップされるため、検証がスムーズになります。 |
| `make deploy` (L4) | `googleapi: Error 400: Precondition check failed.`<br>(detail: `Cloud billing quota exceeded`) | 使用している課金アカウントに紐付けられるプロジェクト数が、GCP側の割り当て上限に達している。 | 1. GCPコンソールで不要なプロジェクトを完全に削除（シャットダウン→即時削除）し、課金アカウントとのリンクを解除する。<br>2. または、別の課金アカウントを用意し `common.tfvars` の `billing_account_id` を更新する。<br>3. **【推奨】** VPCなどの検証が主目的であれば、`python3 terraform/scripts/toggle_test_mode.py` を実行してテストモードを【ON】にしてください。重たい管理プロジェクト（logsink, monitoring等）の作成がスキップされ、課金枠の消費を大幅に抑えることができます。 | コードの不具合ではなくGCP環境側の制約。GCP Free Tierアカウントや、特定の契約下で発生しやすい。 |
| `make deploy` | `Error 403: Project ... has been deleted.`<br>`Error 409: Requested entity already exists`<br>`Cannot destroy project as deletion_policy is set to PREVENT` の連鎖 | **【Ghost State と 削除保護のデッドロック】**<br>`make destroy ALL` を実行せずに、`common.tfvars` の `project_id_prefix` を変更して `make deploy` を実行したため。<br><br>マルチレイヤー構成では、Baseのプロジェクトが再作成されても、上位の Services の Terraform State に「古いプロジェクト（ゴミ箱行き）」の記憶が残ってしまい、アクセス不可（403）や保護ロック（PREVENT）による不整合が発生します。 | **【IaC運用の黄金ルール】**<br>プレフィックスの変更や環境の作り直しを行う場合は、必ず以下の手順を守ってください。<br><br>1. `common.tfvars` のプレフィックスを\*\*「古い名前」のまま\*\*にしておく。<br>2. `allow_resource_destruction = true` に設定する。<br>3. ロック解除を反映させるため、一度 `make deploy` を実行する。<br>4. **必ず `make destroy ALL` を実行し、Stateも実環境も完全に更地にする。**<br>5. 完全に更地になってから、プレフィックスを「新しい名前」に変更する。<br>6. `allow_resource_destruction = false` に戻して `make deploy` を実行する。 | マルチレイヤー特有の罠です。一部のディレクトリを手動で `terraform state rm` するなどの個別対応よりも、この「黄金ルール」に従い、全体を一括で更地にしてから再構築するアプローチが最も確実で安全です。 |
