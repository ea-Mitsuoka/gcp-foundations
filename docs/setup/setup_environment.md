# 複数環境の管理と方針

本基盤では、Terraform の Workspace 機能を使用せず、ディレクトリ分割とスプレッドシート（SSoT）による一元管理を採用しています。

## 1. なぜ Workspace を使わないのか

Workspace は同一のコードで環境（Prod/Dev）を切り替えるのに便利ですが、以下の理由から採用を見送りました。

- **視認性の低下**: 現在どの環境を操作しているか直感的に分かりにくい。
- **環境差異の管理**: プロジェクトごとに異なる API やポリシーの設定を Workspace だけで管理するとコードが複雑化する。

## 2. 本基盤の管理フロー (SSoT ベース)

Workspace の代わりに、以下の仕組みで複数環境・複数プロジェクトを管理します。

1. **ディレクトリによる分離**: 各プロジェクトは `terraform/4_projects/<プロジェクト名>` 配下で独立して管理されます。
1. **Excel による一括定義**: `gcp-foundations.xlsx` ですべてのプロジェクトと環境（Prod/Stag/Dev）を一覧形式で定義します。
1. **自動生成**: `make generate` コマンドにより、Excel の内容から各プロジェクト用の `terraform.tfvars` や自動生成コードが個別に書き出されます。

これにより、**「コードの共通化（テンプレート）」と「環境ごとの柔軟な設定（Excel）」を両立**させつつ、事故の少ない確実なデプロイを実現しています。

詳細は、**[プロジェクトのライフサイクル管理](../operations/project_lifecycle.md)** を参照してください。
