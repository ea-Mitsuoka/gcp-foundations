# ガバナンス・エボリューション・ガイド (運用継続と責務の境界)

本ドキュメントは、本リポジトリ `gcp-foundations` の長期的な運用方針と、中央管理（インフラ基盤チーム）と現場（アプリケーションチーム）の責務の境界線についての指針を定めたものです。

______________________________________________________________________

## 1. 運用の基本理念

本基盤は、組織の成長に合わせて「すべての面倒を見るフルマネージド型」から、プロジェクトを安全に払い出す **「プロジェクト・ベンディング・マシン（払い出し機）」** へと役割を移行させることを前提に設計されています。

「現場の自律性」を尊重しつつ、組織として譲れない「統制（Shared VPC / VPC-SC / 監査ログ）」をどう維持し続けるかが運用の鍵となります。

______________________________________________________________________

## 2. 責務の分離マトリクス (Central vs. Local)

安定稼働後も **「中央（本リポジトリ）」で管理し続けるべきリソース** と、**「現場」へ委譲すべきリソース** を明確に分離します。

| 管理対象 | 責務 | 管理場所 | 理由 |
| :--- | :--- | :--- | :--- |
| **リソース階層 (組織/フォルダ)** | **中央** | `resources` シート | 組織構造の整合性と命名規則の維持。 |
| **プロジェクトの作成/削除** | **中央** | `resources` シート | 課金紐付けと、組織レベルのクォータ管理。 |
| **共有VPCの接続 (Host-Service)** | **中央** | `resources` シート | 共通ネットワークのセキュリティとIP設計の統制。 |
| **VPC-SC (境界防御)** | **中央** | `resources` シート | 組織のデータ境界は、単一プロジェクトの権限で変更させてはならない。 |
| **組織ポリシー (Org Policy)** | **中央** | `org_policies` シート | ガードレールの強制。 |
| **ログ集約 (Sink)** | **中央** | `log_sinks` シート | 監査証跡の改ざん防止と一元管理。 |
| **API 有効化 (ベースライン)** | **中央** | `project_apis` 列 | 共通 NW 接続や監視に必須の API の初期開通。 |
| **API 有効化 (アプリ固有)** | **現場** | 各プロジェクトの IaC | 開発速度を優先し、インフラチームをボトルネックにしない。 |
| **プロジェクト内リソース (VM/GKE等)** | **現場** | 各プロジェクトの IaC | ビジネスロジックの領域。 |

## 3. 運用フェーズの進化 (Evolution Phases)

組織の成熟度に応じて、本リポジトリの運用重心を切り替えます。

### フェーズ 1: 立ち上げ期 (初期構築)

- **状態**: 組織にクラウドの知見が少なく、統制を最優先する時期。
- **管理範囲**: **フルマネージド**。API 有効化も含め、Excel ですべてをコントロールする。
- **ゴール**: 組織の標準となる「正しい GCP 環境」を迅速に量産する。

### フェーズ 2: 安定稼働・拡大期 (ハイブリッド管理)

- **状態**: プロジェクト数が増え、基盤チームが個別の API リクエストを捌ききれなくなる時期。
- **管理範囲**: **プロビジョニングのみ**。Excel では「箱（プロジェクト・ネットワーク・境界）」の準備に徹し、API 有効化の権限（プロジェクト編集者等）を現場に渡す。
- **ゴール**: 「統制」は中央で守りつつ、アプリ構築の自由度を現場に返す。

### フェーズ 3: 成熟期 (プラットフォーム・エンジニアリング)

- **状態**: 現場が独自の Terraform リポジトリを持ち、自律的に開発する時期。
- **管理範囲**: **ガードレール提供**。本リポジトリは「現場が IaC を実行するためのサービスアカウント」と「セキュアな NW 接続」を提供するインターフェースとなる。
- **ゴール**: セキュリティ境界（VPC-SC）の番人としてのみ、中央の Excel を運用し続ける。

______________________________________________________________________

## 4. 運用を継続するための 3 つのポイント

安定稼働後も本リポジトリの価値を維持し、形骸化を防ぐための鉄則です。

### 1. 「境界の番人」であり続ける

現場がどれだけ自由にリソースを作っても、**「どのネットワークに繋がっているか」「どの VPC-SC 境界に守られているか」** の 2 点だけは、必ず resources シートを通じてのみ変更を許可します。これが組織を「無法地帯」にさせない最後の砦です。

### 2. SSoT (Excel) の形骸化を防ぐ

手動でプロジェクトを作ったり、VPC-SC に追加したりすることを固く禁じます。「Excel にないものは組織に存在しない」という文化を徹底します。Excel の 1 行は、インフラチームと現場の間の **「利用契約」** です。

### 3. API 管理の引き際を見極める

`project_apis` 列を無理にすべて埋めようとせず、ある段階で **「基盤として動作を保証する標準セット」** だけを定義するようにシフトし、アプリ固有の API は現場の責任（現場の IaC）に任せることで、基盤チームの運用負荷を最適化します。

## 5. 結論

本リポジトリと Excel (SSoT) は、**GCP 組織の「地図」と「検問所」** です。
API 管理の責務はやがて現場に移譲されるべきですが、**プロジェクトの所属（フォルダ、VPC、VPC-SC）を管理する役割は、その組織が存在し続ける限り中央に留めるべき**です。それが、大規模 GCP 環境においてセキュリティと自由度を両立させる唯一の解だからです。
