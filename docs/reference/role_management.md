# IAM・ロール管理方針 (IAM & Role Management Policy)

本ドキュメントでは、本基盤における権限管理の設計思想、Terraform実行用サービスアカウントの権限、および組織レベルで定義される管理用グループの役割について解説します。

---

## 1. 基本方針 (Core Principles)

本基盤では、Google Cloud のベストプラクティスに基づき、以下の原則に従って権限を管理します。

1. **最小権限の原則 (Least Privilege)**: ユーザーやサービスアカウントには、その役割を果たすために必要な最小限の権限のみを付与します。
2. **グループベースの権限管理**: 個人のアカウントに直接ロールを付与するのではなく、Google グループに対してロールを付与し、ユーザーをグループに参加させることで権限を管理します。
3. **サービスアカウントの借用 (Impersonation)**: 作業者は強力な権限を持つ JSON キーをダウンロードするのではなく、Terraform 専用サービスアカウントを「借用」して作業を行います。

---

## 2. Terraform サービスアカウントの権限

Terraform を実行するサービスアカウント (`terraform-org-manager`) は、インフラ全体を管理するために強力な権限を保持します。

### なぜ強力な権限が必要か？
このサービスアカウントは、組織全体のガバナンス（組織ポリシー、フォルダ構造、共通ログ基盤）を構築・維持する責務を担っているためです。

- **フォルダ・プロジェクトの作成**: 新しいフォルダやプロジェクトを払い出すために `roles/resourcemanager.folderAdmin` や `roles/resourcemanager.projectCreator` が必要です。
- **組織ポリシーの管理**: セキュリティベースラインを強制するために `roles/orgpolicy.policyAdmin` が必要です。
- **組織レベルのログ集約**: 組織全体のログを BigQuery や GCS に送るシンクを作成するために `roles/logging.admin` が必要です。

### 権限の削除について
基盤の初期構築が完了した後でも、**このサービスアカウントの権限を削除することは推奨されません**。新しいプロジェクトの追加や、既存リソースの構成変更（Terraform による State 差分確認）のたびに、これらの権限が必要となるためです。

---

## 3. 管理用 Google グループの定義

`Layer 2: Organization` では、運用担当者の役割に応じた Google グループへの権限付与を一元的に行います。

| グループ名 | 主な役割・権限 |
| :--- | :--- |
| **gcp-organization-admins** | **組織管理者**。組織全体のポリシー、IAM、プロジェクト作成など、すべての管理権限。 |
| **gcp-billing-admins** | **課金管理者**。請求先アカウントの紐付け、コスト確認、課金設定の管理。 |
| **gcp-security-admins** | **セキュリティ管理者**。セキュリティセンターの管理、IAM監査、KMS管理、ログ閲覧。 |
| **gcp-vpc-network-admins** | **ネットワーク管理者**。共有VPC（Shared VPC）の構築、ファイアウォール、NATの設定。 |
| **gcp-logging-monitoring-admins** | **ログ・監視管理者**。ログシンクの構成、監視アラート、ダッシュボードの管理。 |
| **gcp-devops** | **開発運用**。各環境のリソース状況の閲覧やデプロイ支援。 |

---

## 4. ユーザーの作業フロー (Impersonation)

セキュリティを担保するため、管理者は以下の手順で作業を行います。

1. 管理者は自分の個人のアカウントで `gcloud auth login` を行います（権限は非常に限定的です）。
2. Terraform 実行時に、`terraform-org-manager` サービスアカウントを一時的に借用します。
3. これにより、**「誰が」「いつ」「どのサービスアカウントとして」**アクションを起こしたかが監査ログに明確に記録され、鍵の流出リスクも排除されます。

---

## 5. カスタムロールの活用

基本的には Google Cloud プリセットのロールを使用しますが、特定の運用要件（例：90日間未ログインのアカウント検知など）でプリセットロールでは権限が不足、または過剰な場合は、`Layer 2` にてカスタムロールを定義し、最小権限を徹底します。
